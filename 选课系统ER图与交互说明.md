# 选课系统 ER 图与交互说明

## 1. 文档用途

本文档用于对当前选课系统的数据库设计与系统交互过程进行完整说明，适合在课程设计展示、答辩讲解或项目汇报中使用。

本文档主要回答三个问题：

1. 系统的 ER 图是如何设计的
2. 各数据表为什么这样设计，它们之间是什么关系
3. 用户在系统中的完整操作流程是怎样的

当前系统采用的技术路线为：

- 前端：`Streamlit`
- 后端：`Python`
- 数据库连接：`PyMySQL`
- 数据库：运行在本地 Docker 容器中的 `MySQL`
- 运行环境：Conda 虚拟环境 `db_st`

## 2. 系统设计目标

本系统实现的是一个典型的“学生选课成绩管理系统”，其核心目标包括：

- 支持学生登录并进行选课、退课
- 支持学生查询自己的成绩单
- 支持教师查看自己负责的课程班次并录入成绩
- 支持管理员维护学生、教师、课程、开课安排等基础信息
- 支持系统根据角色进行权限控制
- 支持将学生、教师、课程、学期、选课和成绩统一纳入一个完整的数据库结构中管理

系统的设计重点不在复杂算法，而在于：

- 数据库建模是否规范
- 表之间关系是否清晰
- 用户权限是否合理
- 业务流程是否完整

## 3. 为什么要这样设计 ER 图

在一个简单的选课系统中，最容易想到的设计通常只有三张表：

- 学生表
- 课程表
- 选课表

但是这样的结构很快会遇到问题，例如：

- 同一门课程在不同学期开设怎么办
- 同一门课程由不同教师授课怎么办
- 课程有容量限制怎么办
- 选课时间如何控制
- 成绩修改记录如何保存
- 管理员、教师、学生如何统一登录

因此，本系统没有采用“极简三表结构”，而是采用了更完整、更适合真实业务的 ER 设计。

核心思想是：

- 将“课程定义”和“开课安排”分开
- 将“账号信息”和“身份资料”分开
- 将“选课关系”和“成绩数据”合并到选课记录中
- 用日志表记录成绩修改历史

这使得系统结构更加规范，也更适合后续扩展。

## 4. ER 图总体说明

系统围绕以下七类对象展开：

1. 组织结构对象
2. 用户与身份对象
3. 学期对象
4. 课程对象
5. 开课对象
6. 选课对象
7. 成绩与日志对象

从整体上看，ER 图可以概括为下面这条主线：

`院系 -> 专业 -> 学生`

`账号 -> 学生/教师/管理员`

`课程 -> 开课班次 -> 选课记录 -> 成绩`

`学期 -> 开课班次`

`教师 -> 开课班次`

也就是说，学生不是直接选“课程”，而是选“某学期某教师开设的某个课程班次”。这是整个 ER 设计中最重要的一点。

## 5. ER 图中的实体设计

### 5.1 department：院系表

该表用于表示学院或院系信息，例如：

- 计算机学院
- 数学学院

主要字段：

- `dept_id`：院系编号，主键
- `dept_name`：院系名称
- `office_phone`：联系电话
- `office_location`：办公地点

设计原因：

- 教师需要属于某个院系
- 专业需要隶属于某个院系
- 课程也需要有开课院系

因此，院系表是组织结构设计中的根节点。

### 5.2 major：专业表

专业表用于表示学生所属专业。

主要字段：

- `major_id`：专业编号，主键
- `major_name`：专业名称
- `dept_id`：所属院系，外键，指向 `department`

设计原因：

- 一个院系下面通常包含多个专业
- 学生更准确地属于某个专业，而不是直接属于院系
- 后续按专业统计学生人数、班级分布会更方便

例如：

- `CS01`：计算机科学与技术
- `CS02`：软件工程
- `MA01`：数学与应用数学

### 5.3 user_account：统一账号表

这是系统登录的核心表，所有用户都通过这张表登录。

主要字段：

- `user_id`：用户编号，主键
- `username`：用户名，唯一
- `password_hash`：密码哈希
- `role`：角色，取值为 `admin`、`student`、`teacher`
- `status`：账号状态，取值为 `active`、`disabled`
- `last_login_at`：最近登录时间

设计原因：

- 学生、教师、管理员都需要登录系统
- 如果分别给三类人建三张账号表，登录逻辑会变复杂
- 用统一账号表可以让认证逻辑集中，权限判断更清晰

因此，系统采用“统一账号 + 角色区分”的设计。

### 5.4 student：学生表

学生表保存学生的业务资料。

主要字段：

- `student_id`：学号，主键
- `user_id`：账号编号，外键，关联 `user_account`
- `student_name`：学生姓名
- `gender`：性别
- `birth_date`：出生日期
- `enroll_year`：入学年份
- `major_id`：所属专业，外键
- `class_name`：班级名称
- `phone`：电话
- `email`：邮箱
- `status`：学籍状态

设计原因：

- 账号信息和个人资料分离，更规范
- 一个学生对应一个账号，但账号表不应保存学生全部资料
- 学生表是后续选课、成绩查询的核心实体

### 5.5 teacher：教师表

教师表保存教师业务资料。

主要字段：

- `teacher_id`：教师号，主键
- `user_id`：账号编号，外键
- `teacher_name`：姓名
- `gender`：性别
- `dept_id`：所属院系，外键
- `title`：职称
- `phone`：电话
- `email`：邮箱
- `status`：在职状态

设计原因：

- 教师也需要独立业务资料
- 课程开设需要绑定任课教师
- 成绩录入权限要根据教师身份判断

### 5.6 admin_profile：管理员资料表

管理员表保存管理员的扩展资料。

主要字段：

- `admin_id`：管理员编号，主键
- `user_id`：账号编号，外键
- `admin_name`：管理员姓名
- `phone`：联系电话

设计原因：

- 管理员虽然不直接参与选课，但属于系统角色之一
- 保留管理员资料表可以让结构更加统一

### 5.7 semester：学期表

学期表负责管理选课发生的时间范围。

主要字段：

- `semester_id`：学期编号，主键
- `semester_name`：学期名称
- `start_date`：学期开始日期
- `end_date`：学期结束日期
- `selection_start`：选课开始时间
- `selection_end`：选课结束时间
- `status`：状态，如 `planned`、`open`、`closed`

设计原因：

- 选课并不是永久开放的
- 开课记录必须属于某个具体学期
- 成绩单也需要按学期显示

因此，学期表既管理教学周期，也控制选课窗口。

### 5.8 course：课程定义表

这张表表示“课程本身”，而不是具体某一次开课。

主要字段：

- `course_id`：课程号，主键
- `course_name`：课程名称
- `course_type`：课程类型
- `credit`：学分
- `total_hours`：总学时
- `dept_id`：开课院系，外键
- `description`：课程简介
- `status`：课程状态

设计原因：

- 课程本身是长期存在的定义
- 例如“数据结构”这门课可以每学期重复开设
- 所以课程定义不能直接等同于选课对象

### 5.9 classroom：教室表

教室表用于表示上课地点。

主要字段：

- `classroom_id`：教室编号
- `building`：楼宇名称
- `room_no`：房间号
- `capacity`：容量

设计原因：

- 开课班次需要安排上课地点
- 展示系统时，学生看到课程地点会更真实
- 后续也可以扩展为排课管理

### 5.10 course_offering：开课班次表

这是整个系统中最关键的表之一。

它表示：

“某学期，由某教师，在某教室，对某门课程的一次具体开设”

主要字段：

- `offering_id`：开课编号，主键
- `course_id`：课程号，外键
- `semester_id`：学期编号，外键
- `teacher_id`：任课教师号，外键
- `classroom_id`：教室编号，外键
- `max_capacity`：容量上限
- `selected_count`：当前已选人数
- `schedule_text`：上课时间说明
- `status`：开课状态

设计原因：

- 学生真正选择的对象是“开课班次”
- 这样才能区分：
  - 不同学期
  - 不同教师
  - 不同上课时间和地点
  - 不同容量限制

如果没有这张表，系统就无法正确表达真实教学业务。

### 5.11 course_prerequisite：先修课程关系表

用于表示课程之间的先修关系。

主要字段：

- `course_id`：目标课程
- `prereq_course_id`：先修课程

设计原因：

- 某些课程并不是任何学生都能直接选择
- 例如“数据库原理”通常需要先学“数据结构”
- 系统在选课时可以根据这个表做先修判断

### 5.12 enrollment：选课记录表

这张表表示学生与开课班次之间的对应关系。

主要字段：

- `enrollment_id`：选课记录编号，主键
- `student_id`：学号，外键
- `offering_id`：开课编号，外键
- `select_time`：选课时间
- `status`：状态，如 `selected`、`dropped`、`completed`
- `final_score`：最终成绩
- `gpa_point`：绩点
- `remark`：备注

设计原因：

- 学生和开课班次之间是多对多关系
- 需要单独一张关联表来保存选课行为
- 同时这张表还记录成绩与状态，因此它不仅是关联表，也是业务表

系统中很多核心操作都围绕这张表展开：

- 选课
- 退课
- 成绩录入
- 成绩单展示

### 5.13 score_change_log：成绩修改日志表

用于记录成绩修改历史。

主要字段：

- `log_id`：日志编号
- `enrollment_id`：选课记录编号
- `old_score`：旧成绩
- `new_score`：新成绩
- `changed_by_user_id`：修改人账号编号
- `changed_at`：修改时间
- `reason`：修改原因

设计原因：

- 教师和管理员可能修改成绩
- 如果没有日志，后续无法追踪谁改了什么
- 有了日志表后，系统的设计会更严谨，也更适合课程设计展示

## 6. 实体之间的关系说明

### 6.1 组织结构关系

- 一个院系可以包含多个专业
- 一个院系可以包含多个教师
- 一个院系可以开设多门课程
- 一个专业可以拥有多个学生

因此：

- `department -> major`
- `department -> teacher`
- `department -> course`
- `major -> student`

### 6.2 账号与身份关系

- 一个账号可以对应一个学生
- 一个账号可以对应一个教师
- 一个账号可以对应一个管理员

通常一个账号只属于一种角色，因此：

- `user_account -> student`
- `user_account -> teacher`
- `user_account -> admin_profile`

### 6.3 课程与教学关系

- 一门课程可以在多个学期开设
- 一门课程在同一学期也可以存在多个班次
- 一个教师可以承担多个开课班次
- 一个学期可以有多个开课班次
- 一个教室可以安排多个班次

因此：

- `course -> course_offering`
- `teacher -> course_offering`
- `semester -> course_offering`
- `classroom -> course_offering`

### 6.4 选课关系

- 一个学生可以选择多个班次
- 一个班次可以被多个学生选择
- 学生与班次之间的关系通过 `enrollment` 建立

因此：

- `student -> enrollment`
- `course_offering -> enrollment`

### 6.5 成绩关系

- 每条选课记录最多有一个当前最终成绩
- 每条选课记录可以有多条成绩修改日志

因此：

- `enrollment -> score_change_log`
- `user_account -> score_change_log`

## 7. 为什么“课程”和“开课班次”必须分开

这是整个系统建模中最值得单独讲的一点。

如果只保留课程表，那么系统只能表示：

- 课程号
- 课程名
- 学分
- 开课院系

但现实中学生选课时还必须知道：

- 这是哪个学期的课
- 谁来上这门课
- 上课地点在哪里
- 容量还剩多少
- 上课时间是什么

这些信息都不是“课程定义”的属性，而是“开课安排”的属性。

所以本系统采用：

- `course`：描述课程本身
- `course_offering`：描述课程在某学期的一次实际开设

这会带来三个明显好处：

1. 更符合真实教学场景
2. 选课逻辑更清晰
3. 后续扩展更容易

## 8. 数据结构设计的规范性说明

在数据库结构设计中，本系统尽量遵循以下原则：

### 8.1 主键清晰

每张核心表都有明确主键，例如：

- 学生用 `student_id`
- 教师用 `teacher_id`
- 课程用 `course_id`
- 开课班次用 `offering_id`

这样便于唯一标识和表间关联。

### 8.2 外键明确

所有核心关系都通过外键约束表达，例如：

- 学生的 `major_id` 外键到专业表
- 开课班次的 `course_id` 外键到课程表
- 选课记录的 `student_id` 外键到学生表

这有助于保持数据一致性。

### 8.3 避免冗余

例如：

- 用户名和密码不重复存到学生表、教师表中
- 教师信息不直接写进课程表
- 学期信息不直接写进选课表

减少冗余可以降低更新异常。

### 8.4 保证可扩展性

例如：

- 增加教室表后，后续可扩展排课
- 增加日志表后，后续可扩展审计
- 增加先修关系表后，后续可扩展复杂选课规则

## 10. 学生端交互过程

### 10.1 登录

学生在登录页输入：

- 用户名
- 密码

系统会在 `user_account` 表中验证：

- 用户名是否存在
- 密码哈希是否匹配
- 账号状态是否为 `active`

如果验证成功：

- 系统读取账号的 `role`
- 如果角色是 `student`
- 再通过 `user_id` 找到对应的 `student_id`
- 最后把登录状态写入 `st.session_state`

此时学生进入学生端页面。

### 10.2 查看可选课程

学生点击“我的选课”后，系统执行以下逻辑：

1. 读取当前开放学期
2. 查找该学期中状态为 `open` 的开课班次
3. 排除当前学生已经处于 `selected` 状态的班次
4. 计算剩余名额
5. 联合课程表、教师表、教室表，展示完整课程信息

页面上看到的不是抽象课程，而是具体班次，包括：

- 课程号
- 课程名
- 学分
- 课程类型
- 任课教师
- 上课时间
- 上课地点
- 剩余名额

### 10.3 执行选课

学生点击“选课”按钮后，系统会执行一系列校验：

1. 当前学期是否开放选课
2. 当前时间是否处于选课窗口
3. 开课班次状态是否为开放
4. 名额是否已满
5. 是否已经选过该班次
6. 是否曾退课后重新恢复
7. 是否满足先修课程要求

校验通过后，系统在 `enrollment` 中插入一条新记录，状态为 `selected`。

同时，由数据库触发器自动更新：

- `course_offering.selected_count`

这意味着：

- 选课记录保存到 `enrollment`
- 班次人数自动同步到 `course_offering`

### 10.4 执行退课

学生点击“退课”按钮后，系统会检查：

1. 该记录是否确实属于当前学生
2. 当前状态是否为 `selected`
3. 是否已录入成绩
4. 当前学期是否开放退课
5. 当前时间是否仍在选退课窗口内

如果允许退课，则把选课记录状态改为 `dropped`。

数据库触发器会自动把班次人数减一。

### 10.5 查询成绩单

学生点击“我的成绩单”后，系统从 `enrollment` 中查询状态为 `completed` 的记录，并联查：

- `course_offering`
- `course`
- `semester`
- `teacher`

最终展示：

- 学期
- 课程名称
- 课程类型
- 学分
- 任课教师
- 成绩
- 绩点

同时页面会统计：

- 已修门数
- 累计学分
- 平均成绩
- 加权绩点

## 11. 教师端交互过程

### 11.1 登录

教师登录与学生类似，只是登录成功后：

- 角色为 `teacher`
- 系统根据 `user_id` 找到对应 `teacher_id`

进入教师端后，只能访问成绩管理页。

### 11.2 查看自己负责的班次

教师进入成绩管理页面后：

1. 先选择学期
2. 系统只显示当前教师负责的开课班次

这是通过 `teacher_id` 对 `course_offering` 进行筛选实现的。

因此教师看不到其他教师的班次。

### 11.3 查看学生名单

选定某个班次后，系统从 `enrollment` 查询该班次所有未退课学生，并联查学生表，展示：

- 学号
- 姓名
- 班级
- 当前成绩

### 11.4 录入或修改成绩

教师在页面中输入成绩后，系统进行校验：

1. 当前教师是否真的有权维护该班次
2. 成绩是否为空
3. 成绩是否在 `0-100` 范围内
4. 该记录是否已经退课

如果校验通过：

- 系统把 `final_score` 写入 `enrollment`
- 自动计算 `gpa_point`
- 把 `status` 更新为 `completed`
- 同时向 `score_change_log` 写入一条成绩修改日志

### 11.5 查看成绩分布

教师完成录分后，系统会根据该班次学生成绩统计分布，例如：

- 90-100
- 80-89
- 70-79
- 60-69
- 不及格

再用图表展示成绩分布情况。

## 12. 管理员端交互过程

### 12.1 登录

管理员登录后，系统识别角色为 `admin`，进入管理员端页面。

管理员具有最高权限，可以访问：

- 管理员首页
- 学期管理
- 开课安排
- 课程维护
- 学生维护
- 教师维护
- 成绩管理

### 12.2 查看系统概览

管理员首页展示全局统计信息，例如：

- 在籍学生人数
- 在职教师人数
- 开放课程数
- 本期选课数
- 各院系学生人数分布
- 各班次选课情况

这些数据都是通过聚合查询得到的。

### 12.3 维护课程

管理员可以：

- 查看课程列表
- 新增课程
- 修改课程名称、类型、学分、状态等
- 删除没有关联开课班次的课程

课程的维护对象是 `course` 表，也就是课程定义本身。

### 12.4 维护开课安排

管理员可以：

- 选择学期查看当前班次
- 新增班次
- 修改班次的教师、教室、容量、时间、状态
- 删除没有任何选课记录的班次

这部分操作主要影响 `course_offering` 表。

这一步非常关键，因为它把“课程定义”真正转化成“学生可以选择的对象”。

### 12.5 维护学生与教师

管理员可以新增、修改和删除学生、教师。

在新增时，系统会：

1. 先在 `user_account` 中创建账号
2. 再在 `student` 或 `teacher` 表中写入资料

这两步通过事务一起完成，从而避免“只创建账号但资料没写进去”的脏数据问题。

### 12.6 全局成绩管理

管理员还可以像教师一样进入成绩管理页面，但不同的是：

- 教师只能看到自己的班次
- 管理员可以看到全部班次

这体现了管理员的全局权限。

## 13. ER 图如何支撑整个系统运行

如果从系统实现的角度总结，ER 图并不仅仅是画图，它直接决定了以下关键能力：

### 13.1 支撑登录

通过 `user_account` 实现统一认证，通过角色字段实现页面分流。

### 13.2 支撑选课

通过 `course_offering` + `enrollment` 表示学生实际选择的课程班次。

### 13.3 支撑成绩管理

通过 `enrollment.final_score` 保存当前成绩，通过 `score_change_log` 保留修改历史。

### 13.4 支撑统计展示

通过院系、专业、学期、课程、班次等表的关联，可以做：

- 学院分布统计
- 课程人数统计
- 成绩分布统计
- 学生成绩单展示

### 13.5 支撑扩展

未来如果继续扩展，还可以加入：

- 排课冲突检测
- 多教师合授
- 学生培养方案
- 课程类别统计
- 学分预警

这说明当前 ER 图不仅能满足当前项目，也为后续扩展保留了空间。

## 14. 结论

本系统的 ER 图设计具有以下特点：

1. 结构完整，不是简单三表拼接
2. 角色清晰，统一账号管理规范
3. 课程与开课班次分离，符合真实教学场景
4. 选课、成绩、日志三者连接紧密
5. 能够支撑学生、教师、管理员三类角色的完整业务流程

从数据库课程设计的角度来看，这套 ER 设计的价值在于：

- 既体现了数据库建模的规范性
- 又能直接支撑实际系统开发
- 同时具有较强的可展示性和可扩展性

因此，这份 ER 图不仅是数据库结构图，更是整个选课系统业务逻辑的核心基础。
